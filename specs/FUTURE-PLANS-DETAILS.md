## 系統概述

Email Newsletter CMS 系統是一個整合的內容管理與電子報發送平台，專為教育機構（幼稚園/學校）設計，用於管理每週電子報內容並發送給家長。系統的核心價值在於：

- **個人化內容呈現**：根據家長孩子所在班級，動態顯示相關的「班級大小事」
- **統一發送體驗**：一個家長無論有幾個孩子，只收到一封郵件，但能看到所有相關班級內容
- **靈活的內容管理**：支援多媒體內容（圖片、影片、錄音）和文章排序管理
- **數據驅動決策**：整合開信率、點擊率等統計數據，優化內容策略

**現況**: 基本週報閱讀器（唯讀）
**目標**: 具個人化郵件發送功能的完整 CMS

## 完整 User Stories

### Email Newsletter 功能（5 項）

#### US-EMAIL-001: 班級對應的內容區塊 (P1)
**目標**：為每個班級創建獨立的內容區塊，確保專屬內容正確管理和分發

**驗收標準摘要**：
- 管理員可管理班級列表並創建內容區塊
- 文章可指定給特定班級
- 家長只看到所屬班級的專屬內容

---

#### US-EMAIL-002: 每週複製模板並修改內容 (P2)
**目標**：從上週電子報複製模板，快速創建新週報並保持格式一致

**驗收標準摘要**：
- 可複製週報為草稿並編輯內容
- 複製不影響原模板
- 固定內容（如善話）自動填入

---

#### US-EMAIL-003: 開信率與點擊率統計 (P2)
**目標**：追蹤郵件開信率和連結點擊率，了解閱讀習慣並優化策略

**驗收標準摘要**：
- 查看發送數、開信數、開信率
- 追蹤連結點擊並歸類到文章
- 支援分班統計和報表匯出

---

#### US-EMAIL-004: 家長多子女合併電子報 (P1)
**目標**：多孩子家長只收一封郵件，但能看到所有相關班級內容

**驗收標準摘要**：
- 系統自動合併多班級內容到一封郵件
- 網頁同時顯示所有相關班級內容
- 自動調整內容當家長資料更新

---

#### US-EMAIL-005: 統一外觀的個人化郵件 (P2)
**目標**：郵件外觀統一，但根據身份動態顯示個人化內容

**驗收標準摘要**：
- 所有郵件外觀一致（品牌形象）
- 根據 token 或登入狀態顯示專屬內容
- 通用內容與班級專屬內容分離渲染

### 1. 管理後台系統 (優先順序 1)

**功能範圍**：
- **管理儀表板**：內容及用戶管理的中央樞紐
  - 週報清單與狀態追蹤（草稿、已發佈、已歸檔）
  - 文章發布日曆視圖
  - 批量發佈/歸檔操作
  - 發布歷史與審計日誌
- **班級與用戶管理**：完整的組織結構管理
  - 班級列表與編輯（新增、編輯、刪除班級）
  - 班級成員管理（家長、學生關聯）
  - 教師指派（班級專屬編輯員）
  - 批量導入用戶（CSV）
  - 支持雙重角色：同一用戶可同時為教師和家長
- **內容管理介面**：文章的完整生命週期管理
  - 文章分類與標籤管理
  - 文章搜尋與篩選
  - 版本歷史與還原功能
  - 內容回收站（已刪除文章恢復）
- **權限與存取控制 UI**：細粒度的權限管理
  - 角色設置管理
  - 個人化班級限制設定
  - 批量權限調整
  - 存取日誌

**技術考量**：
- 儀表板設計：高效能的資料聚合與視覺化
- 批量操作：優化 UI 效能與資料庫查詢
- 雙重角色處理：角色優先級與衝突解決
- 審計追蹤：所有管理操作的完整記錄

**SpecKit 工作流**：
1. 使用 `/speckit.specify` 創建管理後台功能規格
2. 使用 `/speckit.plan` 設計儀表板架構與資料聚合
3. 使用 `/speckit.tasks` 生成實作任務清單

---

### 2. 分析與報表 (優先順序 2)

**功能範圍**：
- **開信率追蹤**：追蹤每封郵件的開信數量與比例
- **點擊率追蹤**：追蹤文章內連結的點擊次數
- **視覺化儀表板**：圖表呈現統計數據（折線圖、長條圖、圓餅圖）
- **報表匯出**：支援 CSV、Excel 格式下載
- **分班統計**：按班級分組查看數據對比

**技術考量**：
- 整合郵件服務商 API（SendGrid Webhooks、Mailchimp Reports API）
- 圖表庫：Chart.js、Recharts、Apache ECharts
- 資料聚合：定時同步第三方數據到本地資料庫
- 隱私合規：匿名化處理，符合 GDPR/CCPA

**SpecKit 工作流**：
1. 使用 `/speckit.specify` 定義統計指標與儀表板 UI
2. 使用 `/speckit.plan` 設計數據同步與聚合架構
3. 使用 `/speckit.tasks` 分解為 API 整合、前端圖表等任務

### 3. 郵件整合 + 個人化 (優先順序 3)

**功能範圍**：
- **班級內容區塊**：每個班級有獨立的內容區塊
- **多子女合併**：一個家長有多個孩子時，只收到一封郵件
- **動態內容生成**：根據家長的孩子班級動態組合郵件內容
- **唯一追蹤 URL**：每封郵件包含唯一 token 用於追蹤和身份識別
- **模板系統**：可重複使用的郵件模板（HTML + 動態變數）
- **Kit Email Platform（ConvertKit）API 整合**：訂閱者管理與自定義欄位同步
- **Webhook 處理**：接收訂閱、退訂、自定義欄位更新等事件

**技術考量**：
- 郵件服務商：SendGrid（推薦）、Mailchimp、AWS SES
- 模板引擎：Handlebars、EJS
- 批量發送邏輯：避免重複查詢，優化資料庫效能
- 追蹤機制：開信追蹤（Tracking Pixel）、點擊追蹤（Redirect URL）
- **API 選擇與自定義欄位設計**：
  - Kit (ConvertKit)：專注於創作者郵件行銷，強大的自動化功能
  - 自定義欄位：`child_classes`, `child_names`, `parent_type`
  - 標籤策略：每個班級對應一個標籤（如 `class:小熊班`）
- **API 限流處理**：批量操作時遵守 API rate limits
- **錯誤重試機制**：網路失敗時自動重試（指數退避）

**SpecKit 工作流**：
1. 使用 `/speckit.specify` 定義郵件生成邏輯與個人化規則
2. 使用 `/speckit.plan` 設計資料庫查詢優化、模板架構與 API 客戶端
3. 使用 `/speckit.tasks` 創建發送流程、API 整合、Webhook 任務

### 4. 媒體管理 (優先順序 4)

**功能範圍**：
- **媒體庫系統** - 集中管理所有上傳的圖片、音訊、影片
  - 按日期、類型、標籤組織媒體檔案
  - 搜尋與過濾功能（按檔案名稱、上傳日期、尺寸）
  - 媒體預覽（縮圖、音訊播放器、影片預覽）
  - 批量操作（刪除、標籤、匯出）

- **媒體重用與共享** - 在多篇文章間重用媒體
  - 參考計數追蹤（某媒體被使用在幾篇文章）
  - 防止刪除正在使用的媒體
  - 媒體使用歷史（查看某媒體在哪些文章中使用）
  - 快速插入最常用媒體的建議

- **媒體優化與變體** - 為不同用途自動生成媒體變體
  - 自動建立縮圖版本（編輯器瀏覽、郵件預覽）
  - 自動產生 WebP 格式（現代瀏覽器最佳化）
  - 響應式圖片生成（srcset 支援）
  - 音訊轉換（MP3/WAV 互轉，位率最佳化）

- **媒體使用儀表板** - 可視化媒體管理與使用情況
  - 儲存空間使用統計
  - 媒體類型分佈圖
  - 最常用媒體排名
  - 未使用媒體清單（可安全刪除）
  - 媒體大小與格式最佳化建議

**技術考量**：
- 媒體庫 UI：媒體瀏覽器網格/清單視圖、搜尋、篩選、排序
- 媒體使用追蹤：在 `media_usage` 表追蹤每個媒體的引用
- 媒體重用工作流：編輯器中插入現有媒體、媒體庫快速選擇
- 媒體優化管道：圖片自動建立縮圖、WebP 轉換、音訊轉換
- 性能最佳化：高效的查詢、快取策略、CDN 集成

**SpecKit 工作流**：
1. 使用 `/speckit.specify` 定義媒體庫功能與用戶操作流程
2. 使用 `/speckit.plan` 設計媒體儲存架構與優化管道
3. 使用 `/speckit.tasks` 分解為媒體庫 UI、追蹤系統、優化管道等任務

## 關鍵技術決策

### 編輯器類型：所見即所得（TipTap）

**理由**：
- 老師和行政人員可能不熟悉 Markdown 語法
- 圖片、影片嵌入在視覺化編輯器中更直觀
- TipTap 基於 ProseMirror，擴展性強，支援客製化

**替代方案**：
- Quill：輕量但擴展性較弱
- Markdown 編輯器：適合技術用戶，但學習曲線高

---

### 郵件服務：SendGrid / Mailchimp

**理由**：
- 內建追蹤功能（開信率、點擊率）
- 提供模板管理與 WYSIWYG 編輯器
- 高送達率（Deliverability）與信譽管理
- 豐富的 API 與 Webhook 支援

**替代方案**：
- AWS SES：成本更低，但需自建追蹤系統
- Postmark：適合交易型郵件，行銷功能較弱

---

### 資料庫：PostgreSQL

**理由**：
- 複雜的關聯式資料（用戶、學生、班級、文章多對多關係）
- 支援 JSON 欄位（靈活儲存文章內容）
- 強大的查詢優化與索引功能
- 開源且社群活躍

**替代方案**：
- MongoDB：適合非結構化資料，但關聯查詢較弱
- MySQL：功能類似，但 PostgreSQL 在複雜查詢上更優

---

### 身份驗證：Firebase Authentication / Auth0

**理由**：
- 快速整合，減少開發時間
- 支援多種登入方式（Email/Password、Google、Magic Link）
- 內建會話管理與 Token 刷新
- 安全性由專業團隊維護

**替代方案**：
- 自建驗證系統：完全控制但開發與維護成本高
- Supabase Auth：開源替代方案，功能相似

---

### 媒體儲存：Cloudinary / AWS S3

**理由**：
- Cloudinary：自動圖片優化、格式轉換、CDN 整合
- AWS S3：成本較低，適合大量檔案儲存
- 兩者皆支援 CDN 加速，提升全球訪問速度

**建議**：
- 圖片使用 Cloudinary（自動優化）
- 音訊/影片使用 S3（成本考量）

## 資料模型重點

### 核心實體關係

```
User（用戶：管理員/老師/家長）
  ↓ 1:N
Child（學生）
  ↓ N:1
Class（班級：小熊班、小兔班...）
  ↓ 1:N
Newsletter（週報）
  ↓ 1:N
Article（文章含權限控制）
  ↓ 1:N
MediaAttachment（媒體附件：圖片/音訊/影片）
```

### User（用戶）

```typescript
interface User {
  id: string
  email: string
  name: string
  role: 'admin' | 'teacher' | 'parent'
  children?: Child[]        // 家長關聯的孩子
  managedClasses?: Class[]  // 老師負責的班級
  createdAt: Date
  updatedAt: Date
}
```

### Child（學生）

```typescript
interface Child {
  id: string
  name: string
  classId: string
  parentIds: string[]  // 支援多位家長（父母）
  enrolledAt: Date
  isActive: boolean
}
```

### Class（班級）

```typescript
interface Class {
  id: string
  name: string          // 如「小熊班」
  grade: string         // 如「大班」
  teachers: string[]    // 教師 ID 列表
  isActive: boolean
  academicYear: string  // 如「2024-2025」
}
```

### Newsletter（週報）

```typescript
interface Newsletter {
  id: string
  weekId: string        // 如 '2025-W46'
  publishDate: Date
  weeklyQuote: string   // 每週一句善話
  articles: Article[]
  status: 'draft' | 'published'
  createdAt: Date
  updatedAt: Date
}
```

### Article（文章）

```typescript
interface Article {
  id: string
  newsletterId: string
  title: string
  content: string       // HTML 或 Markdown
  order: number         // 排序序號
  visibility: 'public' | 'class_specific'
  classIds: string[]    // 可見的班級 ID（如設為 class_specific）
  author: string        // 作者 ID
  createdAt: Date
  updatedAt: Date
  mediaAttachments?: MediaAttachment[]
}
```

### MediaAttachment（多媒體附件）

```typescript
interface MediaAttachment {
  id: string
  type: 'image' | 'audio' | 'video'
  url: string
  thumbnailUrl?: string
  fileSize: number
  metadata?: {
    duration?: number   // 音訊/影片長度（秒）
    dimensions?: {      // 圖片尺寸
      width: number
      height: number
    }
  }
  uploadedAt: Date
}
```

### EmailCampaign（郵件發送記錄）

```typescript
interface EmailCampaign {
  id: string
  newsletterId: string
  sentAt: Date
  totalRecipients: number
  deliveredCount: number
  openedCount: number
  clickedCount: number
  bouncedCount: number
  recipientGroups: {
    classId: string
    recipientCount: number
  }[]
}
```

### Analytics（統計數據）

```typescript
interface Analytics {
  articleId: string
  views: number
  uniqueVisitors: number
  clicks: number           // 文章內連結點擊次數
  avgReadTime: number      // 平均閱讀時間（秒）
  clicksByLink: {
    url: string
    clicks: number
  }[]
  lastUpdated: Date
}
```
